## 4. custom backend 配置

Mossen 个人版默认不走官方 OAuth / 官方账号登录，而是走 custom backend / OpenAI-compatible 后端。

本机当前配置文件：

```bash
.mossensrc/custom-backend.env
```

这个文件是本地私密配置，包含真实 API key，不要提交、不要外发、不要粘贴到公开文档。

`run-bun-featured.sh` 会自动加载它。关键变量如下：

```bash
export MOSSEN_CODE_USE_CUSTOM_BACKEND=1
export MOSSEN_CODE_CUSTOM_BASE_URL="https://your-openai-compatible-endpoint/v1"
export MOSSEN_CODE_CUSTOM_NAME="Your Backend Name"
export MOSSEN_CODE_CUSTOM_MODEL="your-model-id"
export MOSSEN_CODE_CUSTOM_API_KEY="your-api-key"
export MOSSEN_CODE_CUSTOM_BACKEND_PROTOCOL="openai-compatible"
```

也可以用 token：

```bash
export MOSSEN_CODE_CUSTOM_AUTH_TOKEN="your-token"
```

或者自定义 headers：

```bash
export MOSSEN_CODE_CUSTOM_HEADERS='{"Authorization":"Bearer xxx"}'
```

修改后端的方法：

```bash
cd /Users/allen/Documents/aiproject/mossensrc
vim .mossensrc/custom-backend.env
./run-mossen.sh
```

临时覆盖某个变量：

```bash
cd /Users/allen/Documents/aiproject/mossensrc
MOSSEN_CODE_CUSTOM_MODEL="another-model" ./run-mossen.sh
```

如果启动后看到类似 “No Mossen backend is configured”，说明 custom backend 环境变量没有加载成功，先检查 `.mossensrc/custom-backend.env` 是否存在。

如果看到 `api.mossen.invalid` 或 `platform.example` 相关提示，通常说明某个 hosted 能力没有配置。这不影响个人版主线；个人版默认不依赖官方 hosted 服务。

### 4.1 多模型 profile 管理 (S1-09)

Mossen 支持多个模型 profile 并存（qwen / minimax / glm 等），通过 `~/.mossen/settings.json` 内的 `mossen.profiles` 字段管理；优先级 active profile > 旧 `MOSSEN_CODE_CUSTOM_*` env > 默认。

**9 个 CLI flag**（任何阶段都可在终端跑，立即生效）：

```bash
# 列出所有 profile + 当前 active（apiKey 自动脱敏）
mossen --list-model-profiles

# 查看某个 profile（无参 = 当前 active）
mossen --get-model-profile [<name>]

# 创建新 profile
mossen --add-model-profile minimax \
  --provider openai-compatible \
  --baseURL https://api.minimaxi.com/v1 \
  --model MiniMax-M2.7 \
  --apiKey sk-cp-xxx \
  --name "MiniMax 2.7"

# 部分更新（只改你给的字段，其他保留）
mossen --update-model-profile minimax --model MiniMax-M3.0

# 单独换 apiKey（高频用法）
mossen --set-model-profile-key minimax sk-new-xxx

# 切换当前 active profile（mossen 启动时用它）
mossen --set-model-profile minimax

# 测试 profile 连通性（真发 GET <baseURL>/models 验 reachable + apiKey 透传）
mossen --test-model-profile minimax [--timeout 5000]

# 删除 profile（如果它是当前 active，自动清掉 active）
mossen --delete-model-profile minimax

# 把旧 env fallback (MOSSEN_CODE_CUSTOM_*) 一次性升级为正式 settings profile
# - 默认 name=qwen, scope=user, activate=auto (不动已显式选的 active)
# - --force 覆盖同名已有 profile; --activate always 强制设为 active
mossen --migrate-fallback-profile [--name qwen] [--force] [--activate auto|always|never]
```

**settings.json 示例**（实际由 CLI 写入，不要手编辑）：

```json
{
  "mossen.profiles": {
    "qwen": {
      "provider": "openai-compatible",
      "baseURL": "https://coding.dashscope.aliyuncs.com/v1",
      "model": "qwen3.6-plus",
      "apiKey": "sk-sp-xxx",
      "name": "Qwen 3.6 Plus"
    },
    "minimax": {
      "provider": "openai-compatible",
      "baseURL": "https://api.minimaxi.com/v1",
      "model": "MiniMax-M2.7",
      "apiKey": "sk-cp-xxx"
    }
  },
  "mossen.activeProfile": "qwen"
}
```

**安全要求**：
- `~/.mossen/settings.json` 包含真 apiKey，文件权限建议 `chmod 600`
- 所有 CLI dump / log / 错误输出必须脱敏 apiKey（前 6 + ... + 后 4，e.g. `sk-cp-...wxyz`）
- 不允许把真 apiKey 写入 git tracked 文件
- 没有 active profile 时，自动 fallback 到旧 `.mossensrc/custom-backend.env`，qwen 默认零破坏

**与旧 env 的关系**：旧 `MOSSEN_CODE_CUSTOM_*` 仍然兼容（D-S09-3=P），但定位从主路径降为**迁移兜底**。

- 有任意 settings profile → fallback **不进入** `/model` 列表和 `--list-model-profiles allProfiles[]`（避免成为正常主路径）。`fallbackProfile` 字段仍反映 env 真实存在性，供 UI 检测迁移机会。
- settings 完全为空 + env 存在 → fallback 兜底进入列表，并在 `/model` 中提示用户运行 `mossen --migrate-fallback-profile` 升级为正式 profile。
- 一次性运行 `mossen --migrate-fallback-profile` 即可把 qwen 从 fallback 升级为正式 `mossen.profiles.qwen`，与 glm/minimax 平等管理。`.mossensrc/custom-backend.env` 文件不删，老启动方式继续可用。

## 6. 全局命令是怎么设置的

### 6.1 当前机器现状

当前这台机器上，`mossen` 还没有进入 PATH：

```bash
which mossen
```

当前会显示找不到。

同时机器上有旧的 `claude` 命令：

```bash
which claude
```

当前指向：

```bash
/Users/allen/.local/bin/claude
```

这是旧官方二进制，不是 Mossen 源码启动入口。后续开发 Mossen 时不要依赖这个 `claude` 命令。

### 6.2 推荐方式：用 npm link 创建全局 mossen 命令

`package.json` 已声明：

```json
"bin": {
  "mossen": "./run-mossen.sh"
}
```

所以可以在项目目录执行：

```bash
cd /Users/allen/Documents/aiproject/mossensrc
npm link
```

然后检查：

```bash
which mossen
mossen --help
```

这会把全局 `mossen` 命令链接到本项目的 `run-mossen.sh`。

如果以后项目目录移动了，重新执行：

```bash
cd /新路径/mossensrc
npm link
```

如果要取消全局链接：

```bash
npm unlink -g mossen-code-src
```

### 6.3 备选方式：手动软链接

如果不想用 `npm link`，可以手动创建软链接：

```bash
mkdir -p "$HOME/.local/bin"
ln -sfn /Users/allen/Documents/aiproject/mossensrc/run-mossen.sh "$HOME/.local/bin/mossen"
```

确保 `~/.local/bin` 在 PATH 里：

```bash
echo "$PATH" | tr ':' '\n' | grep -x "$HOME/.local/bin"
```

如果没有，把下面这一行加入 `~/.zshrc`：

```bash
export PATH="$HOME/.local/bin:$PATH"
```

然后重新打开终端，或执行：

```bash
source ~/.zshrc
```

检查：

```bash
which mossen
mossen --help
```

如果以后项目目录移动了，重新设置软链接：

```bash
ln -sfn /新路径/mossensrc/run-mossen.sh "$HOME/.local/bin/mossen"
```

### 6.4 不建议覆盖 claude 命令

不要把 `claude` 强行覆盖成 Mossen，除非你明确知道自己在做什么。

如果确实为了兼容旧习惯要临时让 `claude` 调 Mossen，建议只在当前 shell 里做临时 alias：

```bash
alias claude='mossen'
```

不要默认写进全局配置。否则以后排查时会分不清是官方二进制还是 Mossen 源码。

## 11. 官方 hosted 能力的边界

当前 Mossen 个人版不依赖官方账号、官方 OAuth、官方 hosted marketplace、官方 Chrome 扩展、官方远程工作区。

默认主线：

- custom backend
- OpenAI-compatible 模型接口
- 本地 CLI/TUI
- skills
- MCP
- hooks
- plugins
- worktree
- 项目记忆
- 状态栏
- 上下文压缩

如果未来要接 hosted 服务，应通过 Mossen 自己的 adapter 做，不要直接恢复旧官方默认路径。

可能需要的 hosted 环境变量包括：

```bash
export MOSSEN_CODE_PLATFORM_BASE_URL="https://your-mossen-platform.example"
export MOSSEN_CODE_ENABLE_HOSTED_AUTH_ADAPTER=1
```

Chrome/browser hosted URL 示例：

```bash
export MOSSEN_CODE_ENABLE_CHROME=1
export MOSSEN_CODE_CHROME_EXTENSION_URL="https://your-platform/chrome"
export MOSSEN_CODE_CHROME_FOCUS_TAB_URL_BASE="https://your-platform/chrome/tab/"
export MOSSEN_CODE_CHROME_PERMISSIONS_URL="https://your-platform/chrome/permissions"
export MOSSEN_CODE_CHROME_RECONNECT_URL="https://your-platform/chrome/reconnect"
export MOSSEN_CODE_CHROME_DOCS_URL="https://your-platform/docs/chrome"
```

个人版当前不需要配置这些。

## 12. 常见问题

### 12.1 `mossen: command not found`

说明还没有设置全局命令。

推荐：

```bash
cd /Users/allen/Documents/aiproject/mossensrc
npm link
```

或手动软链接：

```bash
mkdir -p "$HOME/.local/bin"
ln -sfn /Users/allen/Documents/aiproject/mossensrc/run-mossen.sh "$HOME/.local/bin/mossen"
```

### 12.2 启动时提示找不到旧目录

如果看到：

```text
当前工作目录缺失
```

通常是某个会话还绑定在已经不存在的旧路径。现在正确目录是：

```bash
/Users/allen/Documents/aiproject/mossensrc
```

重新从新目录打开会话即可。

不建议为了兼容旧会话创建额外软链接；直接使用当前目录即可。

### 12.3 `Cannot find module '@ant/claude-for-chrome-mcp'`

这通常说明你启动的不是当前新仓库源码，或拿到了包含官方内部浏览器集成包静态依赖的历史源码。

解决方式：

- 确认工作目录是 `/Users/allen/Documents/aiproject/mossensrc`。
- 确认当前分支是 `main`。
- 当前 Mossen 源码已把该能力改成可选 adapter，不应该因为缺私有包导致主程序启动失败。

### 12.4 后端连接失败

先检查：

```bash
sed -n '1,80p' .mossensrc/custom-backend.env
```

确认至少有：

```bash
MOSSEN_CODE_USE_CUSTOM_BACKEND=1
MOSSEN_CODE_CUSTOM_BASE_URL=...
MOSSEN_CODE_CUSTOM_MODEL=...
MOSSEN_CODE_CUSTOM_API_KEY=...
```

如果使用 token 或 headers，确认对应变量存在。

### 12.5 登录没用或 `/login` 被禁用

这是设计如此。个人版不需要官方 OAuth。

应通过 custom backend 环境变量配置模型后端，而不是登录官方账号。

### 12.6 Chrome 集成不可用

当前个人版默认禁用 Chrome 集成。

如果看到：

```text
Chrome 集成未配置
```

这不是核心启动失败。除非你已经有 Mossen 自己的浏览器扩展和 hosted URL，否则先忽略。

### 12.7 状态栏 ctx 不准

检查 `~/.mossen/statusline-command.sh`，确保优先读：

```bash
.context_window.used_percentage
```

不要优先读：

```bash
.context_observability.pressure_percent
```
